# システム開発におけるAIエージェント活用計画

## 1. 目的

本計画は、システム開発工程でAIエージェントをどこに適用し、誰が何を実施し、どの記録を残すかを明確にするための説明資料である。

メンバー向けには日々の作業手順を分かりやすく示し、開発PL向けには品質担保の仕組みと責任分界を明確にすることを目的とする。

## 2. 基本方針

| 項目 | 方針 |
|---|---|
| 作成責任 | 成果物の最終責任は人が持つ |
| Agent整備 | AI推進補佐が各工程用Agentを作成する |
| Agent実行 | 作業者またはAI推進者が用途に応じてAgentを指定して実行する |
| レビュー記録 | すべてのレビュー系Agentはレビュー記録を必ず出力する |
| 品質基準 | Agentには工程ごとのチェックリストを持たせる |
| 全体整合 | 全体共通観点はAI推進者が全体レビューで一括確認する |
| 人のレビュー | セルフレビューは作業者、成果物レビューはレビュアーが実施する |

## 3. 役割分担

| 役割 | 主な責任 |
|---|---|
| AI推進補佐 | 各工程の作成Agent、レビューAgent、品質管理Agentを作成し、チェックリストを定義する |
| 作業者 | 成果物を作成し、指定されたAgentを使って個別レビューとセルフレビューを実施し、修正を反映する |
| AI推進者 | 全体レビューAgentを実施し、工程内の横断的な不整合を一括修正する |
| レビュアー | 工程品質管理の成果物レビューを実施する |
| 開発PL | 実施ルール、品質担保、記録出力の運用状況を確認する |

## 4. 共通運用ルール

| ルール | 内容 |
|---|---|
| レビュー記録の保存 | Agent実行ごとにレビュー記録を出力し、成果物と同じ単位で保存する |
| チェックリスト管理 | 各Agentには対象成果物に対応したチェックリストを持たせる |
| 全体共通レビュー観点 | 表紙、目次、変更履歴、コピーライトなどの共通項目は全体レビューAgentで確認する |
| 是正方式 | 個別指摘は作業者が修正し、全体整合の指摘はAI推進者が一括修正する |
| 品質管理 | 工程品質管理としてセルフレビューと成果物レビューを必ず実施する |

## 5. 全体フロー

| Step | 実施内容 | 主担当 | 出力 |
|---|---|---|---|
| 1 | AI推進補佐が工程別Agentとチェックリストを準備する | AI推進補佐 | Agent定義、チェックリスト |
| 2 | 作業者が成果物を作成する | 作業者 | 設計書、実装、テスト設計書 |
| 3 | 作業者が個別レビューAgentを実行し、一次品質を整える | 作業者 | レビュー記録、修正版成果物 |
| 4 | AI推進者が全体レビューAgentを実行し、横断観点を一括確認する | AI推進者 | 全体レビュー記録、一括修正版 |
| 5 | 作業者がセルフレビューを実施する | 作業者 | セルフレビュー記録 |
| 6 | レビュアーが成果物レビューを実施する | レビュアー | 成果物レビュー記録 |

## 8. 開発PL向けの具体確認ポイント

開発PLは「AIを使っているか」ではなく、「品質が担保される運用になっているか」を確認する。確認時は、以下の観点ごとに成果物と記録を見ればよい。

| 観点           | 開発PLが確認すること                              | 確認するファイル                | 判断基準                                                  |
| ------------ | -------------------------------------- | ----------------------- | ----------------------------------------------------- |
| 1. 作成責任の明確化  | 成果物の最終更新者と修正責任者が人になっているか               | 各工程の成果物本体、作成記録          | Agentが下書きや点検をしていても、最終版の確定者が作業者またはAI推進者になっている          |
| 2. 個別レビュー実施  | 成果物ごとにAgentレビューが実施され、指摘と修正結果が残っているか    | 個別レビュー記録、成果物本体          | すべての成果物にレビュー記録があり、重大指摘が未対応のまま残っていない                   |
| 3. 全体レビュー実施  | 表紙、目次、変更履歴、コピーライト、文書間整合の確認が横断的に行われているか | 全体レビュー記録                | AI推進者が全体レビューを実施し、横断指摘に対する修正結果が残っている                   |
| 4. セルフレビュー実施 | 作業者自身がセルフレビューを実施しているか                  | セルフレビュー記録               | 作業者名、実施日、確認観点、自己判定、修正有無が記録されている                       |
| 5. 成果物レビュー実施 | 第三者レビューが実施されているか                       | 成果物レビュー記録               | レビュアー名、判定、指摘事項、承認可否が記録されている                           |
| 6. チェックリスト運用 | Agentごとのチェックリストが定義され、レビュー観点に漏れがないか     | Agent定義、チェックリスト、各レビュー記録 | 設計書ガイドライン、コーディングルール、テスト観点がレビュー記録に反映されている              |
| 7. 記録の追跡性    | どの版に対して誰がいつレビューしたか追跡できるか               | すべての記録ファイル              | 成果物名、Gitコミット参照（版数はGit管理）、実施日、実施者、判定がファイル名または本文に含まれている |
| 8. 工程横断の再現性  | 基本設計から結合テスト設計まで同じ運用ルールで回せるか            | 工程別一式の成果物と記録            | 工程が違っても、作成記録、レビュー記録、セルフレビュー記録、成果物レビュー記録の構成が揃っている      |

### 8.1 開発PLが最低限見るべき順番

| 手順  | 確認内容           | 確認目的                      |
| --- | -------------- | ------------------------- |
| 1   | 成果物本体を確認する     | 何が納品対象かを把握する              |
| 2   | 個別レビュー記録を確認する  | 作業者がAgentで一次品質を確保しているかを見る |
| 3   | 全体レビュー記録を確認する  | 文書間の整合や共通項目の漏れがないかを見る     |
| 4   | セルフレビュー記録を確認する | 作業者が自己点検責任を果たしているかを見る     |
| 5   | 成果物レビュー記録を確認する | 第三者レビューで承認可能な状態かを見る       |

### 8.2 開発PLが見るときの具体的な質問

| 質問 | 見るポイント |
|---|---|
| この成果物は誰が最終責任を持ったか | 作成記録と最終更新者 |
| Agentレビューは実施したか | 個別レビュー記録の有無 |
| 横断的な体裁不備や整合不備は潰したか | 全体レビュー記録の指摘と修正結果 |
| 作業者は自分で見直したか | セルフレビュー記録の有無と内容 |
| 第三者が承認できる状態か | 成果物レビュー記録の判定 |
| 後から監査や説明ができるか | ファイル名、Gitコミット履歴、実施日、実施者の記録 |

## 9. 期待効果

| 効果      | 内容                                     |
| ------- | -------------------------------------- |
| 品質の平準化  | チェックリストをAgentに持たせることでレビュー観点の抜け漏れを減らせる  |
| 作業効率化   | 作業者はAgentを指定して実行することで、作成とレビューの初動を早められる |
| 横断整合の向上 | AI推進者が全体レビューを一括実施することで文書間の不整合を減らせる     |
| 説明責任の強化 | すべてのレビューで記録を残すため、開発PL説明や監査対応がしやすい        |

## 10. 導入時の補足

導入初期は、工程ごとにチェックリスト粒度を合わせることが重要である。特に基本設計、詳細設計、テスト設計では文書品質基準を明文化し、開発ではコーディングルールとユニットテスト通過条件を明確化してから運用を開始する。

また、Agentが出力するレビュー記録のフォーマットを統一しておくと、工程横断での比較、是正管理、開発PL報告がしやすくなる。

### 11.4 開発PLが標準出力物で確認すべきこと

| ファイル | 確認ポイント |
|---|---|
| 成果物本体 | 納品対象として内容が成立しているか |
| 個別レビュー記録 | Agentレビューの指摘・修正結果と作成経緯が分かるか |
| 全体レビュー記録 | 横断整合と共通項目の確認結果が分かるか |
| セルフレビュー記録 | 作業者が自己点検責任を果たしたか |
| 成果物レビュー記録 | レビュアーが承認可否を判断したか |
